# 要件定義書（Claude Code 指示書）再作成
# アプリ名（仮）：Quick Health Logger
# 技術指定：Ruby on Rails / Bootstrap
# 目的：スマホで“開くたびに”サクッと入力→AWSに蓄積→Athenaで分析できる状態にする（可視化は作らない）
# インフラ：Terraformで構築、GitHub Actionsでデプロイ
# 重要：Webアプリだがスマホアプリのように扱う（PWA想定）

============================================================
1. 背景・目的
------------------------------------------------------------
- 日々の主観データ（気分・疲れ・やる気など）と行動フラグ（水分/コーヒー/休憩など）を、スマホを開いたタイミングで何度も入力できるようにする。
- 入力データは、FitbitなどのヘルスケアデータやCO2濃度等の環境データと後で結合して分析するために使う。
- 本アプリは「収集」に特化する。グラフ/ダッシュボード等の可視化は範囲外。

============================================================
2. ゴール（最重要）
------------------------------------------------------------
- スマホでホーム画面から起動できる（PWA）こと
- “起動→入力→送信”が最短で完了し、頻回入力を苦にしないこと
- 入力データがAWS側に蓄積され、Athenaからクエリ可能であること
- Terraform + GitHub Actionsで環境構築とデプロイが再現可能であること

============================================================
3. ユースケース・利用体験（PWA/スマホ最適化）
------------------------------------------------------------
- ユーザーはスマホでアプリ（PWA）を開く
- その場の状態を数秒で入力し送信する（1日に複数回）
- 入力は「今（now）」が基本だが、必要なら任意時刻/任意日付入力も可能（要件として保持）
- オフライン時は一時保存し、オンライン復帰時に送信（PWAの強みとして推奨）

UX要件（頻回入力向け）
- 1画面完結（入力フォームのみ）
- 0-100はスライダー＋数値表示＋微調整（+/-）
- 0/1はトグル
- 前回入力値の再利用（プリセット）を用意（例：直近値を初期値として表示）
- 送信後の遷移は極小（成功トースト表示、同画面維持 or 自動リセット）
- 端末の“ホームに追加”前提の導線を用意（初回のみ）

============================================================
4. 入力項目要件（頻回入力向けスキーマ）
------------------------------------------------------------
「1レコード = 1ユーザー × 1入力イベント（タイムスタンプ単位）」
※日次1回ではなく、1日に何度も入力する前提に変更

必須カラム
- user_id: string（認証から取得）
- recorded_at: timestamp（入力イベントの時刻。基本は送信時刻=nowだが、ユーザーが編集できても良い）
- timezone: string（例: Asia/Tokyo）
- client_id/device_id: string（任意。分析で端末差分が欲しければ）
- app_version: string（任意。後で不具合分析用）

0-100（整数）例
- fatigue_score: int (0-100)
- mood_score: int (0-100)
- motivation_score: int (0-100)

0/1フラグ例（boolean受け取り→内部0/1でもOK）
- drank_water_flag
- drank_coffee_flag
- took_break_flag

任意
- note: string（最大長を決める、例 280）
- tags: string配列相当（Railsではカンマ区切り文字列でも可。保存形式はAthenaで扱いやすく）

拡張性（項目追加が起きる前提）
- 初期は「列持ち（fatigue_score等）」でOK
- ただし将来、項目が増えるので以下のどちらかを採用して理由をREADMEに残すこと：
  A) 列持ち + JSONB(またはtext JSON) の拡張カラム extra_metrics
  B) 縦持ち（metric_key, metric_value）テーブル
- MVPはA（列持ち＋extra_metrics）を推奨：UIと分析の両方が楽

============================================================
5. データ保存・Athena登録要件（最重要）
------------------------------------------------------------
前提
- 最終的にAthenaで分析できる形でS3に蓄積し、Glue Data Catalogにテーブル登録する。

保存先（推奨アーキテクチャ：二段構え）
1) オンライン処理（Webアプリ側）
   - まずRDS(PostgreSQL)に保存（頻回入力＋即時応答＋PWAオフライン同期と相性が良い）
2) 分析用出力（バッチ/ストリーミング）
   - 定期的にS3へParquetでエクスポートし、Athena/Glueで参照

S3の形式（推奨）
- フォーマット：Parquet
- パーティション：dt=YYYY-MM-DD（recorded_atのローカル日付で切る）
- 小ファイル対策：一定時間/一定件数でまとめて書く（例：15分 or 100件）
- データレイアウト例：
  s3://<bucket>/health_logs/dt=2026-02-25/part-0001.parquet

Glue/Athena
- Glue Data Catalogにテーブルを作成
- スキーマ変更ポリシー（列追加は許可、nullable運用、バージョン列追加も検討）

============================================================
6. Railsアプリ要件（機能）
------------------------------------------------------------
画面
- / (New Record) 1ページのみ（頻回入力用）
- 設定画面（任意）：オフライン同期の状態、直近送信、ログアウト等

API
- POST /records
  - バリデーション（0-100、フラグ正規化、note長）
  - DB保存
- GET /records/latest（任意）
  - 前回値をUI初期値に使う
- POST /sync（任意）
  - オフラインキューをまとめて送信する場合

PWA要件
- manifest.json を提供
- Service Worker（Workbox等）でキャッシュ戦略
- オフライン入力キュー
  - ブラウザ側IndexedDBに貯める（推奨）
  - オンライン復帰時に自動送信
- “ホームに追加”導線（iOS/Android差分は簡易でOK）

UI（Bootstrap）
- モバイルファーストのフォーム
- スライダーは0-100（ユーザー要望に一致）
- トグル/チェックは指一本で操作しやすい余白
- 送信ボタンは画面下固定（操作性優先）

------------------------------------------------------------

- PoC/個人用の簡易案：メール/パスワード（Devise）でも可
- Claude Codeは「本番寄り = Cognito」を推奨し、代替案も併記すること

8. インフラ要件（Terraform）
------------------------------------------------------------
前提
- Terraformで再現可能にする（dev/prod分離）
- GitHub Actionsで plan/apply とアプリデプロイを自動化

必要リソース（推奨構成案を提示して選定）
A案（推奨：運用バランス）
- VPC（public/private）
- ALB
- ECS Fargate（Railsコンテナ）
- RDS PostgreSQL
- S3（Athena用）
- Glue（Catalog/Job）
- EventBridge（エクスポート定期実行）
- IAM（最小権限）
- CloudWatch Logs

B案（より簡単）
- AWS App Runner（Railsコンテナ）
- RDS PostgreSQL
- S3 + Glue + EventBridge

C案（サーバーレス志向だがRailsは工夫が必要）
- Lambda + API Gateway（Railsをそのままは重いので非推奨。採用するなら理由が必要）

注意（ユーザーが「ウェブサーバーは特に考えていない」点の扱い）
- “Webアプリを公開して使う”ためにはホスティング先が必須なので、Claude Codeは上の案から最小運用で実現できる案を選び、理由を書いた上でTerraform化すること。
- MVPはB案（App Runner）またはA案（ECS Fargate）。どちらが望ましいかは、既存運用・将来拡張・コストで判断。判断根拠をREADMEに残すこと。

10. 分析用エクスポート（DB→S3→Athena）
------------------------------------------------------------
方式（推奨）
- 定期実行（EventBridge）でバッチを起動
- バッチの実行先：ECSタスク / Glue Job / Lambda（規模に応じて）
- DBから前回以降のレコードを抽出し、ParquetでS3へ書き出す
- Glueテーブルはパーティション追加（dt）を自動化
- 小ファイルを避けるため、時間窓でまとめる（例：15分/1時間）

代替（最小MVP）
- 最初はS3にJSON Linesで保存し、後でParquet化を追加しても良い
- ただし、最終的にはParquet＋partitionに寄せる

============================================================
11. 非機能要件
------------------------------------------------------------
- 可用性：個人/小規模。高可用性は必須ではないが、送信失敗→再送が確実にできること
- 性能：頻回入力（1日10〜50回程度）でもストレスなく動く
- セキュリティ：HTTPS、最小権限、ログに個人情報を過剰出力しない
- コスト：Athenaを前提にParquet/partition、小ファイル対策を考慮

============================================================
12. 受け入れ基準（DoD）
------------------------------------------------------------
- PWAとしてホーム画面に追加でき、スマホアプリのように起動できる
- 起動後すぐ入力でき、1日に何度も記録できる（recorded_at単位で複数件）
- オフライン時は入力を保持し、オンライン復帰で送信される（最低限、手動再送でも可）
- DBに保存される
- 定期エクスポートでS3にParquetが生成され、Glue/AthenaでSELECTできる
- Terraformで一式構築でき、GitHub Actionsでデプロイできる
