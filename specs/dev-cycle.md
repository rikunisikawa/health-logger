あなたはこのリポジトリの自律型開発基盤を設計・実装するリードエンジニア兼システムアーキテクトです。

目的:
Claude Code を用いて、プロダクト開発のサイクルを自動で回す仕組みを、この既存リポジトリに実装してください。
cron で定期起動し、各ロールのエージェントにタスクを割り振り、成果物を蓄積し、最後に人間向けの説明資料として PowerPoint または動画の元データを自動生成する構成にしてください。

最終的に実現したいこと:
- cron または定期実行の仕組みから開発サイクルを起動できる
- オーケストレーターが各ロールエージェントへタスクを割り振る
- Product Manager, Project Manager, Designer, Engineer, Reviewer, Release/Explainer などの役割が分離されている
- タスク状態が構造化され、追跡可能である
- 成果物が docs や tasks 配下に蓄積される
- サイクル完了時に、説明用のスライド原稿を生成できる
- 可能なら pptx 自動生成まで行う
- 動画自体は必須ではないが、後から拡張できる構成にする

前提:
- 既存リポジトリを活かして最適化すること
- 実装は段階的に行うこと
- まずは最小実用構成でよい
- 過剰に複雑なマルチエージェント構成にはしないこと
- 保守しやすいことを重視すること
- 人間が途中で確認・承認できるポイントを残すこと
- 危険な自動デプロイや無制限ループは避けること

最重要の設計方針:
1. cron は単なる起動トリガーにする
2. 実際の采配は orchestrator が行う
3. 各ロールは責務を厳密に分離する
4. タスクは構造化データで管理する
5. 成果物はファイルとして repo に残す
6. 説明資料生成は開発サイクルの最後に必ず通す
7. 後から GitHub Actions にも移植しやすい構成にする
8. hooks は補助に留め、主制御はコードで持つ
9. 将来的な Skills / MCP / subagents 拡張を想定する
10. まずは「動く最小版」を作る

実装してほしい全体構成:
以下のような責務分離を前提として設計・実装してください。

ロール一覧:
- orchestrator
  - サイクル全体の起動、状態遷移、タスク割当、再試行制御、停止条件管理
- product-manager
  - 課題整理、PRD/feature brief 作成、優先順位付け
- project-manager
  - タスク分解、依存関係整理、進行管理
- designer
  - 画面要件、UX観点、文言・状態設計
- engineer
  - 実装、テスト、リファクタ、技術ドキュメント更新
- reviewer
  - 受け入れ条件チェック、レビュー、品質判定
- explainer
  - 人間向け説明資料作成、スライド原稿生成、pptx 出力

必須成果物:
このタスクでは、単なる説明ではなく、実際にリポジトリへ必要ファイルを追加・修正してください。
最低限、以下を作成または更新してください。

1. ディレクトリ構成の提案と実ファイル追加
候補:
- .claude/agents/
- .claude/skills/
- automation/
- tasks/
- docs/product/
- docs/design/
- docs/engineering/
- docs/qa/
- docs/explain/

2. エージェント定義ファイル
各ロールごとに md ファイルを作成し、以下を含めてください。
- 役割
- 入力
- 出力
- 禁止事項
- 成功条件
- 他ロールとの境界

3. タスク管理フォーマット
tasks 配下で扱うタスクのテンプレートを作成してください。
JSON か YAML のどちらかで統一してください。
最低限、以下の項目を持たせてください。
- task_id
- title
- goal
- role
- priority
- status
- dependencies
- inputs
- constraints
- definition_of_done
- outputs
- created_at
- updated_at

4. orchestrator 実装
automation 配下に orchestrator の実装を追加してください。
言語はこの repo に最も自然なものを選んでください。
必要なら Python を優先してください。

orchestrator に必要な機能:
- tasks/inbox, ready, doing, review, done の状態管理
- 今回実行対象タスクの選定
- role ごとの実行ディスパッチ
- エラー時の再試行制御
- 最大実行回数や停止条件
- 実行ログ出力
- dry-run モード
- 人間承認待ち状態のサポート

5. 定期実行スクリプト
cron から叩けるエントリポイントを作成してください。
例:
- automation/run_cycle.sh
- automation/run_cycle.py

6. explainer 実装
開発サイクルの最後に説明資料を生成する流れを作ってください。
最低限:
- 変更内容の要約
- 背景
- 何を作ったか
- なぜ作ったか
- 使い方
- 今後の課題
を含むスライド原稿を JSON または YAML で出力してください。

7. PowerPoint 出力
可能であれば python-pptx を使って pptx を出力する機能を追加してください。
最低限:
- タイトルスライド
- 概要
- 背景
- 実装内容
- 使い方
- 次アクション
の 5〜7 枚程度の構成で出力してください。

8. 設定ファイル
必要に応じて .claude/settings.json を追加または更新し、
- hooks
- permissions
- project 共通設定
を整理してください。
ただし hooks に依存しすぎない構成にしてください。

9. ドキュメント
README または docs 配下に、この自動開発サイクルの使い方を説明するドキュメントを追加してください。
最低限:
- 何をする仕組みか
- ディレクトリ構成
- タスクの流れ
- 実行方法
- cron 設定例
- 安全策
- 今後の拡張ポイント
を含めてください。

実装優先順位:
以下の順番で進めてください。

Phase 1:
- ディレクトリ構成追加
- role agent 定義ファイル追加
- tasks テンプレート追加
- orchestrator の最小実装
- dry-run 可能な run_cycle エントリポイント追加
- explainer の原稿 JSON 出力追加
- README 追加

Phase 2:
- pptx 自動生成追加
- review / approval ステータス強化
- ロギング改善
- 失敗時の再試行制御改善

Phase 3:
- 動画化しやすいナレーション原稿出力
- GitHub Actions への移植を意識した構成整理
- 将来の MCP / Skills 連携ポイントの整理

設計上の制約:
- いきなり本番デプロイするような処理は入れない
- 無限ループを防ぐこと
- 1サイクルあたりの実行上限を設けること
- role が勝手に責務外のファイルを大きく変えないようにすること
- 変更は追跡可能にすること
- 既存構成を尊重しつつ、無理があれば改善案を提示すること
- 実装前に現状の repo 構成を確認し、最も自然な配置に寄せること
- 不要な抽象化や過剰設計は避けること

実装時の具体要求:
- まず repo 全体を確認し、既存の構成・使用言語・運用方針を把握する
- そのうえで「どこに何を置くか」の設計方針を簡潔に整理する
- 次に、最小差分で必要ファイルを追加・修正する
- 各ファイルには目的が分かるコメントまたは記述を入れる
- 実行可能なサンプルタスクを 1〜2 個追加する
- dry-run で一連の流れを確認できるようにする
- 実装後に、変更内容・構成・今後の拡張案をまとめる

期待するアウトプット形式:
以下の順で出してください。

1. 現状把握
- 現在の repo 構成の要約
- 採用する実装方針
- 追加/修正対象ファイル一覧

2. 実装
- 実際にファイルを追加・修正する

3. 実装結果まとめ
- 追加した仕組みの要約
- 実行手順
- cron 設定例
- 制約と今後の改善ポイント

explainer の出力仕様:
explainer は最低限、以下の JSON 形式でスライド原稿を出力できるようにしてください。

{
  "title": "説明資料タイトル",
  "summary": "全体要約",
  "slides": [
    {
      "title": "背景",
      "bullets": [
        "箇条書き1",
        "箇条書き2"
      ],
      "narration": "このスライドで説明する内容"
    }
  ]
}

PowerPoint を作る場合の要件:
- python-pptx を使用してよい
- デザインはシンプルでよい
- テキスト主体でよい
- 図表は必須ではない
- 後から差し替えやすい構成にする
- 出力先は docs/explain/ または artifacts/ 配下にする

成功条件:
- リポジトリに自動開発サイクルの骨組みが追加されている
- 各ロールの責務が明文化されている
- run_cycle で一連の流れを dry-run できる
- explainer により説明資料原稿が生成される
- 可能なら pptx も出力される
- README を読めば運用イメージが分かる
