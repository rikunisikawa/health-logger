以下は Claude にそのまま渡せるように、要件定義書 + 実装方針 + 追加指示をまとめたプロンプトです。

あなたはシニアデータエンジニア、AWSアーキテクト、dbtアーキテクトです。
以下の要件に基づいて、外部の気象・環境データを Web API から取得し、AWS 上の分析基盤に取り込み、個人のヘルスケアデータと関連分析できる仕組みを設計・実装してください。

この依頼では、単なるサンプル実装ではなく、現在のプロジェクト構成にできるだけ適合する形で最適化してください。
既存のディレクトリ構成、命名規則、IaC運用、環境変数管理、ログ方針、テスト方針、データレイクのレイヤー設計、既存のAWSリソース設計がある場合は、それを優先して踏襲してください。
新しい構成を強引に持ち込まず、既存構成に自然に統合できる形を第一優先にしてください。

また、実装だけでなく、事前準備が必要な API、Secrets、AWS リソース、認証情報、環境変数、外部サービス契約事項も一覧化してください。
さらに、データ変換は dbt Fusion を前提とし、必要な変換・正規化・集約・分析用テーブル作成は dbt で実装してください。
dbt 関連の設計・実装・モデル生成・依存関係整理・テスト設計では dbt の MCP サーバーを利用する前提で進めてください。

==================================================
1. 背景・目的
==================================================

個人のヘルスケアデータと、外部の環境データを時系列で統合し、体調や行動との関連性を分析したい。

対象となる環境データの例:
- 気圧
- 気温
- 天気
- 降水
- 湿度
- 花粉量
- 必要に応じて風速、AQI、PM2.5、紫外線など

分析したい内容の例:
- 気圧変動と体調悪化の関係
- 気温と睡眠や活動量の関係
- 花粉量と症状の関係
- 天気や降水と行動量の関係

第一段階のゴールは、これらの外部データを Web API から継続的に取得し、AWS の分析可能なテーブルへ蓄積し、個人ヘルスケアデータと結合できる状態にすること。
高度な因果推論や機械学習本番化は本スコープ外とする。

==================================================
7. AWS アーキテクチャ要件
==================================================

推奨最小構成:
- EventBridge: 定期実行
- Lambda: API 取得処理
- S3: Raw データ保管、および Athena 参照対象データ保管
- Glue Data Catalog: テーブル定義管理
- Athena: クエリ
- dbt Fusion: 正規化、集約、マート作成
- Secrets Manager または SSM Parameter Store: APIキー管理
- CloudWatch Logs: ログ管理

基本フロー:
1. EventBridge が定期的に Lambda を起動
2. Lambda が対象 API を呼び出す
3. レスポンス JSON を S3 raw レイヤーに保存
4. 必要に応じて JSON/Parquet 化した中間データを S3 stage レイヤーへ配置
5. dbt Fusion が stage / intermediate / mart を構築
6. Athena から分析可能にする

重要:
- 変換処理は原則 dbt で行う
- Lambda 側には取得・保存・最低限の正規化以外を持ち込まない
- 既存プロジェクトで Glue Job, ECS, Step Functions, GitHub Actions などが既に使われている場合は、その構成への適合を優先する

==================================================
8. データレイク / テーブルレイヤー要件
==================================================

レイヤー構成:
- raw
- stage
- intermediate
- mart

raw:
目的:
- APIレスポンス原本の保存
要件:
- JSON をそのまま保存
- 再処理可能
- source_name、date、hour、location などでパーティションを切る

stage:
目的:
- API 差異の吸収
要件:
- 列名・型・単位の標準化
- dbt source / staging モデルで吸収

intermediate:
目的:
- 分析しやすい中間特徴量作成
要件:
- 気圧変化量
- 前日との差分
- 直近3時間平均
- 日次統合のための加工

mart:
目的:
- ヘルスケアデータと結合する最終分析用テーブル
要件:
- 時間粒度テーブル
- 日次粒度テーブル
- user/date/hour/location ベースで結合しやすい設計

==================================================
10. 外部 API 要件
==================================================

機能要件:
- HTTPS 接続
- 緯度経度または地点指定で取得可能
- 時間単位または日単位データ取得可能
- 過去データ再取得可能
- JSON 形式
- provider ごとの差異を吸収できる実装にする

設計要件:
- provider interface を切ること
- Open-Meteo を第一候補として組み込むこと
- ただし WeatherAPI, Tomorrow.io 等に差し替え可能な設計にすること
- 取得クライアントと変換ロジックは分離すること

==================================================
11. 取り込み頻度要件
==================================================

- 初期は毎時または日次のいずれかで設定可能にする
- 推奨は毎時取得
- バックフィル実行のため、対象日や期間を指定して再取得できること
- 冪等性を意識し、同一時刻・同一地点・同一ソースの重複登録を防ぐこと

==================================================
12. データ品質要件
==================================================

- 必須カラム欠損時は異常として記録
- 型変換失敗はログ出力
- 異常値チェックを行う
  - 気温: -50〜60℃
  - 気圧: 800〜1100hPa
  - 湿度: 0〜100%
  - 花粉量: 0以上
- 単位を統一する
- 取得元時刻と保存時刻を保持する
- 重複排除ロジックを持つ
- データ品質テストは dbt tests で定義する
- 重要カラムには not_null / unique / accepted_range 相当のチェックを用意する

==================================================
13. セキュリティ要件
==================================================

- API キーは Secrets Manager または Parameter Store で管理
- Lambda には最小権限 IAM を付与
- S3 は暗号化
- ログに秘密情報を出さない
- 個人ヘルスケアデータと結合する際の識別子管理を明確にする
- 既存プロジェクトのセキュリティ方針に従うこと

==================================================
14. 可観測性・運用要件
==================================================

- CloudWatch Logs で開始/終了/失敗を記録
- 構造化ログ(JSON形式推奨)
- 失敗時に再実行可能
- リトライ設計を持つ
- 連続失敗時の通知導線を考慮する
- ingestion_log を通じて実行履歴を追跡できること
- backfill 実行方法を README に明記すること

==================================================
15. 非機能要件
==================================================

保守性:
- provider ごとに client を分離
- 設定値はコード直書きしない
- 既存フォルダ構成に沿って整理
- README と運用手順を明確にする

拡張性:
- API の追加が容易
- 項目追加に耐えられる
- 複数地点対応へ拡張しやすい

性能:
- 単一地点・毎時取得で短時間に処理完了
- 小規模データ取得を前提

信頼性:
- raw 保存により再処理可能
- 一時的 API 障害時も復旧しやすい

18. 実装方針
==================================================

必須方針:
- Python で実装
- AWS Lambda で API 取得
- EventBridge でスケジュール実行
- S3 に raw JSON を保存
- 必要に応じてステージング用データを S3 に保存
- Glue Catalog / Athena を利用
- データ変換・正規化・集約は dbt Fusion で実装
- dbt の MCP サーバーを用いて dbt モデル設計・生成・改善を行う
- Terraform による IaC を基本とする
- pytest で単体テストを用意する
- provider interface を実装する
- backfill 実行パラメータを実装する
- ロケーション、API種別、取得頻度などは設定可能にする
- 構造化ログを採用する
- 既存のプロジェクト構成に最大限合わせる

重要な追加指示:
- 現在のプロジェクト構成を確認し、それに合わせて最適なディレクトリ配置、モジュール分割、Terraform module 構成、dbt project 配置、CI/CD 統合方式を提案・実装すること
- 既存の coding style, lint, formatter, test framework, environment management があれば尊重すること
- もし既存構成が不明な場合は、汎用案を提示した上で、後から既存構成へ差し替えやすい形にすること
