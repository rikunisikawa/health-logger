# AI PM ガバナンスルール

## 基本原則

### 1. 人間が最終承認者

AI エージェントは「提案・分析・草案作成」のみを行う。
以下は **必ずユーザーの明示的な承認** が必要:

| 操作 | 承認が必要な理由 |
|------|----------------|
| スプリント計画の確定 | 工数見積もりの確認が必要 |
| Issue の新規作成・クローズ | 仕様への影響確認が必要 |
| PR のマージ | 品質・影響範囲の最終確認が必要 |
| アーキテクチャ変更 | 非機能要件への影響確認が必要 |
| リリースノートの公開 | ユーザー向け情報の精査が必要 |

### 2. GitHub を Single Source of Truth とする

- ローカルのメモや口頭の合意より GitHub Issue/PR の状態を優先する
- 仕様変更は必ず Issue または PR に記録する
- 設計決定は `docs/ai-pm/adr/` に記録する（将来拡張）

### 3. AI が自律的に行ってはいけないこと

```
❌ 仕様変更の実施（CLAUDE.md・settings.json の変更）
❌ settings.json の deny リストを緩める
❌ ユーザー未承認の Issue/PR 作成
❌ main ブランチへの直接コミット
❌ terraform apply の自律実行
❌ GitHub MCP 経由の破壊的操作（Issue クローズ・PR マージ）
❌ セキュリティ設定（CORS・IAM）の無断変更
```

## エージェント責務マトリクス

| 操作 | pm-agent | integration-agent | architect-agent | engineer-agent | qa-agent |
|------|----------|-------------------|-----------------|----------------|----------|
| Issue 分類・優先度付け | 判断 | 実行 | - | - | - |
| スプリント計画立案 | 判断 | 実行 | 助言 | - | - |
| 技術設計の起案 | - | - | 起案 | 実装 | レビュー |
| コードの実装 | - | - | - | 実行 | - |
| テスト・品質確認 | - | - | - | - | 実行 |
| PR レビューコメント | - | 投稿 | - | - | 判断 |
| ドキュメント更新 | 指示 | - | 執筆 | 執筆 | - |

## GitHub MCP 操作ルール

### 読み取り（承認不要）

- Issue/PR の一覧取得・詳細取得
- マイルストーン一覧取得
- CI/CD 実行状況確認

### 書き込み（ユーザー確認必要）

- Issue コメントの投稿
- ラベルの付与・変更
- マイルストーンの作成・Issue への割り当て
- PR へのレビューコメント投稿
- 新規 Issue の作成

### 禁止（AI は実行不可）

- Issue/PR のクローズ・削除
- PR のマージ
- ブランチの削除
- リポジトリ設定の変更

## フォールバックルール

GitHub MCP が利用できない場合:

1. `gh CLI` でフォールバックする
2. それも使えない場合はユーザーに手順を提示して手動実行を依頼する
3. いかなる場合も「MCP が使えないから操作をスキップ」はしない

## Backlog MCP 拡張予定

将来的に Backlog MCP を追加する場合:

- `.mcp.json` に Backlog サーバー設定を追加
- 各スキルの `providers/backlog.md` を実装
- `CLAUDE.md` の「GitHub MCP 利用ルール」に Backlog の扱いを追記
