# セキュリティ設計

> 対象読者: 業務担当者・プロジェクトオーナー
> 技術的な実装詳細よりも「何を・なぜ守っているか」を中心に記載する

---

## 1. 認証（Authentication）― 「あなたは誰か」を確認する仕組み

### 採用方式: Cognito Hosted UI + OAuth 2.0 PKCE

```
ユーザー → AWS Cognito（AWSが管理するログイン画面）→ アプリ
```

**ポイント:**
- パスワードはアプリ側に渡らない。AWS Cognito のみが管理する
- ログイン画面自体が AWS のサーバーで動いているため、パスワードの盗取リスクが低い
- ログイン成功後に「JWT トークン（デジタル身分証）」がアプリに渡される

### JWT トークンの有効期限

| トークン種別 | 有効期限 | 用途 |
|------------|---------|------|
| ID Token | 1時間 | ユーザー情報の確認 |
| Access Token | 1時間 | API 呼び出しに使用 |
| Refresh Token | 30日 | アクセストークン再発行 |

> **業務上の注意**: 1時間でトークンが切れると再ログインが必要になる場合がある。
> 現在は Amplify ライブラリが自動でリフレッシュを行っている。

## 2. 認可（Authorization）― 「何にアクセスできるか」を制御する仕組み

### データアクセス制御

```
ユーザー A のデータ → ユーザー A のみが取得・更新・削除できる
ユーザー B は ユーザー A のデータにアクセスできない
```

**実装:**
- JWT トークンに含まれる `sub`（ユーザーID）を Lambda が取り出し、自分のデータのみを操作する
- API Gateway がトークンの正当性を検証してから Lambda を呼び出す
- 不正なトークンは API Gateway の時点で 401 エラーとして拒否される

### API エンドポイントの保護

| エンドポイント | 認証必須 | 説明 |
|-------------|---------|------|
| POST /records | ✅ | 記録の作成 |
| GET /records/latest | ✅ | 記録の取得 |
| DELETE /records/{id} | ✅ | 記録の削除 |
| GET /env-data/latest | ✅ | 環境データ取得 |

すべてのエンドポイントに認証が必要。未認証リクエストは即時拒否。

## 3. 通信の暗号化

| 通信経路 | 暗号化 | 方式 |
|---------|-------|------|
| ブラウザ ↔ Amplify (フロントエンド) | ✅ | HTTPS (TLS 1.2+) |
| ブラウザ ↔ API Gateway | ✅ | HTTPS (TLS 1.2+) |
| Lambda ↔ Firehose / S3 | ✅ | AWS 内部通信（暗号化済み） |
| S3 保存データ | ✅ | S3 サーバーサイド暗号化 |

## 4. CORS（クロスオリジンリソース共有）

**CORS = 「どのサイトからの API 呼び出しを許可するか」の設定**

```
許可リスト（cors_allow_origins）:
  ✅ https://main.d24eyg8x5429ma.amplifyapp.com（本番サイト）
  ❌ その他すべてのサイトからのアクセスを拒否
```

> **リスク**: `*`（全許可）に設定すると、悪意のあるサイトから API が呼び出せてしまう。
> 現在は本番サイトのドメインのみに制限済み。

## 5. 入力バリデーション（型チェック）

### なぜ重要か

```
悪意のある入力例:
  fatigue_score: 999    → 0〜100 の範囲外
  note: <script>...</script>  → XSS 攻撃
  user_id: ' OR 1=1 --  → SQL インジェクション
```

### 多層防御の仕組み

```
[レイヤー1] フロントエンド（TypeScript）
  → 型定義（interface）で不正な型を送信できないようにコーディング段階で防ぐ
  → ただし、開発者ツールからの直接送信は防げない

[レイヤー2] Lambda（Pydantic バリデーション）
  → サーバー側でも必ず検証する（フロントだけに頼らない）
  → fatigue_score: 0〜100 の範囲外 → 400 エラー
  → note: 280文字超 → 400 エラー

[レイヤー3] Athena クエリの UUID 検証
  → SQL に埋め込む user_id を正規表現で検証
  → SQL インジェクション対策
```

## 6. シークレット管理

| シークレット | 保存場所 | 管理方法 |
|------------|---------|---------|
| VAPID 秘密鍵（プッシュ通知） | GitHub Secrets | CI/CD 経由で Lambda 環境変数に注入 |
| Cognito クライアントシークレット | Terraform State（S3） | Terraform が自動管理 |
| AWS アクセスキー | 使用しない | GitHub OIDC で一時クレデンシャルを取得 |

> **禁止事項**: シークレット値をコード・PR・チャットに記載しない（過去に VAPID 鍵を PR に記載してしまった事故あり → 即時鍵再生成で対応）

## 7. セキュリティ上のリスクと現状の対応状況

| リスク | 深刻度 | 現状 | 対応状況 |
|-------|-------|------|---------|
| 不正ログイン | 高 | Cognito が管理（MFA 未設定） | △ MFA は任意 |
| データ漏洩（通信） | 高 | 全通信 HTTPS | ✅ |
| データ漏洩（保存） | 高 | S3 暗号化済み | ✅ |
| SQL インジェクション | 高 | UUID 正規表現検証済み | ✅ |
| XSS | 中 | React のデフォルト保護 + 危険な API 不使用 | ✅ |
| CSRF | 中 | JWT Bearer 認証のため無効 | ✅ |
| 過剰なデータアクセス | 中 | user_id でフィルタリング | ✅ |
| 未認証アクセス | 高 | 全 API に JWT 必須 | ✅ |
