# 非機能要件定義

> 非機能要件とは「何をするか（機能）」ではなく「どのように動くか（品質）」を定義するもの。
> パフォーマンス・可用性・スケーラビリティ・保守性・セキュリティなどが含まれる。

---

## 1. パフォーマンス（応答速度）

### 目標値

| 操作 | 目標応答時間 | 現状 | 備考 |
|------|------------|------|------|
| ページ読み込み | 3秒以内 | ✅（CDN配信） | Amplify CDN キャッシュ |
| 記録の送信（POST） | 2秒以内 | ✅（Lambda コールドスタートを除く） | — |
| 最新記録の取得（GET） | 5秒以内 | △（Athena ポーリング最大10秒） | ★後述 |
| ダッシュボード表示 | 10秒以内 | △（データ量依存） | — |

### ★ Athena の遅延について

```
Athena はサーバーレス SQL エンジンのため、クエリ実行に時間がかかる:
  - 初回クエリ: 3〜8秒
  - データ量が増えると比例して遅くなる可能性がある
  - 現状: 最大10秒でタイムアウト（get_latest Lambda の設定）

将来的な改善候補:
  - DynamoDB に最新レコードをキャッシュする
  - Athena クエリ結果キャッシュを活用する
```

### Lambda コールドスタート

```
Lambda は一定時間呼び出しがないと「スリープ」状態になる。
次の呼び出し時に「起動時間（コールドスタート）」が発生:
  - Python 3.13: 通常 500ms〜2秒
  - 日常的に使うアプリなら問題になりにくい
```

## 2. 可用性（稼働率）

### AWS サービスの SLA

| サービス | AWS 公式 SLA | 年間ダウンタイム上限 |
|---------|------------|-------------------|
| API Gateway | 99.95% | 約4.4時間/年 |
| Lambda | 99.95% | 約4.4時間/年 |
| S3 | 99.99% | 約53分/年 |
| Amplify Hosting | 99.9% | 約8.8時間/年 |
| Cognito | 99.9% | 約8.8時間/年 |

> **現状の課題**: 冗長構成なし（シングルリージョン: ap-northeast-1 のみ）。
> AWS 東京リージョン障害時はサービス全体が停止する。
> 個人用アプリのため、現時点では許容範囲とする。

### オフライン対応（PWA）

```
ネットワーク切断時でも記録できる仕組みがある:
  - IndexedDB（ブラウザのローカルDB）にキューイング
  - オンライン復帰時に自動で送信（useOfflineQueue フック）
  - ただし、データ確認（ダッシュボード）はオフラインでは不可
```

## 3. スケーラビリティ（拡張性）

### 現在の設計での上限目安

| リソース | 現在の想定 | 上限の目安 |
|---------|----------|-----------|
| 同時ユーザー数 | 1〜数人（個人利用） | Lambda: 数百〜数千（自動スケール） |
| 記録件数 | 数百〜数千件/年 | S3: 実質無制限 |
| Athena クエリ | 数件/日 | 並列実行数に上限あり |

### スケールアップが必要になるケース

```
現在の設計で問題が出るとしたら:
  1. ユーザー数が数百人規模になった場合
     → Athena の同時クエリ数の上限に達する可能性
     → DynamoDB キャッシュ層の追加を検討

  2. 記録データが数十万件を超えた場合
     → Athena のスキャン量・コストが増加
     → Iceberg パーティション設計の見直しが必要

  3. ダッシュボードへのアクセスが集中した場合
     → Lambda の同時実行数上限（デフォルト1000）に注意
```

## 4. 保守性・運用性

### デプロイの自動化

```
コードを GitHub の main ブランチにマージするだけで:
  1. Lambda のビルド・S3 アップロード（GitHub Actions）
  2. Terraform による infrastructure 更新
  3. Amplify によるフロントエンドのビルド・配信

手動作業は terraform apply の確認のみ（安全のため）
```

### ログの確認方法

```
Lambda の実行ログ:
  AWS マネジメントコンソール → CloudWatch → Log Groups
  → /aws/lambda/health-logger-prod-create-record
  → /aws/lambda/health-logger-prod-get-latest

エラーが発生した場合はここでスタックトレースを確認できる
```

### 監視の現状と課題

| 監視項目 | 現状 | 改善候補 |
|---------|------|---------|
| Lambda エラー率 | ❌ アラート未設定 | CloudWatch Alarm を追加 |
| API Gateway 5xx エラー | ❌ アラート未設定 | CloudWatch Alarm を追加 |
| Athena クエリ失敗 | ❌ アラート未設定 | CloudWatch Alarm を追加 |
| S3 ストレージ使用量 | ❌ 未監視 | S3 Storage Lens を活用 |

> ⚠️ **現状の課題**: エラーが発生しても通知が来ない。
> 定期的に CloudWatch でログを手動確認する運用が必要。

## 5. データ保持・バックアップ

### 現状

| データ | バックアップ | 保持期間 |
|-------|------------|---------|
| 健康記録（S3 Iceberg） | ❌ S3 バージョニング未設定 | 無期限（削除しない限り） |
| Terraform State | ✅ S3 バージョニング有効 | バージョン履歴あり |
| ユーザー情報（Cognito） | AWS が管理 | AWS に依存 |

> ⚠️ **リスク**: S3 Iceberg テーブルのデータをバージョニングなしで運用中。
> 誤ってバケットを削除すると全データが失われる。
> 改善候補: S3 バージョニングの有効化、または別バケットへのクロスリージョンレプリケーション。

## 6. セキュリティ（非機能要件の観点）

→ 詳細は `docs/security.md` を参照

| 項目 | 要件 | 現状 |
|------|------|------|

| データ暗号化（保存時） | S3 暗号化 | ✅ |
| 入力検証 | サーバー側で必ず実施 | ✅ |
| シークレット管理 | コードに書かない | ✅ |

## 7. コンプライアンス・プライバシー

→ 詳細は `docs/cost-management.md` のプライバシーセクションを参照

```
健康データは個人情報の中でも特に機密性が高い「要配慮個人情報」に相当する。
現状は個人利用のため法的義務は限定的だが、以下を意識する:

  - データの収集目的を明確にする（自分の健康管理のみ）
  - 第三者へのデータ提供は行わない
  - ユーザーデータの削除要求に応じられる仕組みがある（DELETE API）
```

## 8. 非機能要件の優先度まとめ

| 項目 | 優先度 | 現状評価 | 対応の緊急度 |
|------|-------|---------|------------|
| セキュリティ（認証・暗号化） | 最高 | ✅ 対応済み | — |
| データ整合性（バリデーション） | 高 | ✅ 対応済み | — |
| パフォーマンス（応答速度） | 中 | △ Athena 遅延あり | 低（個人利用） |
| 可用性 | 中 | △ 単一リージョン | 低（個人利用） |
| 監視・アラート | 中 | ❌ 未設定 | 中（早期に設定推奨） |
| バックアップ | 高 | ❌ 未設定 | 高（データ保護のため） |
| スケーラビリティ | 低 | ✅ サーバーレスで自動対応 | — |
