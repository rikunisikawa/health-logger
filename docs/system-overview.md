# Health Logger — システム概要書

**対象読者**: プログラミング仕様を知らない方でも、このアプリが「何をしていて・どう守られていて・どうデータが流れるか」を把握できるように記述しています。

**最終更新**: 2026-03-14

---

## 1. このアプリの目的

毎日の主観的な体調（疲労感・気分・やる気）と、頭痛・腹痛などの状態、運動・飲酒などの行動を、スマートフォンからすばやく記録するアプリです。

- 1日に何度でも記録できます（「今の状態」を随時スナップショット）
- 蓄積したデータは後から分析できるよう AWS 上に保管されます
- 将来的には Fitbit などのウェアラブルデータや気圧・気温などの環境データと組み合わせて傾向分析に使う想定です

---

## 2. 利用者とスコープ

| 項目 | 内容 |
|------|------|
| 利用者 | 個人（現在は1人） |
| 利用環境 | スマートフォンのブラウザ（ホーム画面に追加してアプリのように使用） |
| 公開範囲 | 非公開（ログインしないと何もできない） |
| 運用環境 | AWS の本番環境のみ（開発環境は常時稼働していない） |

## 3. 記録できる項目

### 3-1. 体調記録（1日の総まとめ的な記録）

| 項目 | 種類 | 説明 |
|------|------|------|
| 疲労感 | 0〜100のスライダー | 0が元気、100が疲れ果てた |
| 気分 | 0〜100のスライダー | 0が最悪、100が最高 |
| やる気 | 0〜100のスライダー | 0がゼロ、100が全開 |
| メモ | テキスト（最大280文字） | 自由記述 |

### 3-2. ステータス（現在進行中の状態）

一瞬で終わるものではなく、「今この状態が続いている」ことを記録します。

| 項目 | 説明 |
|------|------|
| 頭痛 | 頭痛が起きている |
| 腹痛 | 腹痛がある |
| 眠い | 眠気が強い |
| 勤務中 | 仕事中である |

### 3-3. クイックイベント（その瞬間に起きたこと）

ボタンを一度押すだけで即座に記録されます。

| 項目 | 説明 |
|------|------|
| 睡眠不足 | 昨夜の睡眠が足りなかった |
| 運動 | 運動をした |
| アルコール | お酒を飲んだ |
| カフェイン | コーヒー等カフェインを摂取した |

### 3-4. カスタム項目

設定画面から自分で項目を追加できます。スライダー・数値・テキスト・チェックボックスの4種類に対応。体調記録フォームへの追加、クイックイベント、ステータスのいずれかに分類できます。

## 4. ログイン・認証の仕組み

### 4-1. ログイン方式

**Amazon Cognito** というAWSの認証サービスを使っています。

| 項目 | 設定内容 |
|------|---------|
| ログイン方法 | メールアドレス + パスワード |
| パスワード要件 | 8文字以上・大文字・数字・記号を含む |
| 二段階認証（MFA） | オプション（TOTP: Google Authenticator等） |
| メール確認 | 新規登録時に必須（確認メールのリンクをクリック） |

### 4-2. ログインの流れ

```
① ユーザーがアプリを開く
      ↓
② 未ログインの場合、Cognitoのログイン画面へ自動転送
      ↓
③ メール・パスワードを入力してログイン
      ↓
④ ログイン成功 → アプリ本体に戻り「トークン」が発行される
      ↓
⑤ 以降のAPI通信にはすべてこのトークンが自動付与される
      ↓
⑥ トークンは1時間で期限切れ → 自動更新（30日間有効なリフレッシュトークンで更新）
```

**パスワードは通信中に生のままでは送られません。** SRP（Secure Remote Password）という方式により、パスワードの証明はできても実際の文字列は流れない安全な認証プロトコルを使用しています。

### 4-3. トークンについて

ログイン成功後に発行される「IDトークン（JWT）」は、ユーザーの本人確認証明書のような役割を持ちます。

- **有効期限**: 1時間
- **自動更新**: リフレッシュトークン（30日間有効）で自動延長
- **API呼び出しのたびに検証**: サーバー側でAWSが署名を確認し、偽造されたトークンは弾かれます

## 5. データの流れ（データリネージ）

ユーザーが送信ボタンを押してからデータが蓄積されるまでの流れです。

```
【スマートフォン】
  ユーザーが入力→送信ボタン押下
      │
      │ HTTPS（暗号化通信）
      ▼
【API Gateway】
  - トークンが本物かAWSが自動検証
  - 不正なトークンはここでブロック
      │
      ▼
【Lambda（サーバーレス処理）】
  - 入力データのチェック（後述）
  - ユーザーIDをトークンから取得（自称不可）
  - レコードIDを自動採番（UUID）
  - 受信日時を自動付与
      │
      ▼
【Firehose（配送バッファ）】
  - データを一時的に溜めてまとめてS3に流す
  - バッファ条件: 5MB 溜まるか 60秒経過したら流す
  - 通信エラー時はリトライして確実に届ける
      │
      ▼
【S3 Tables（データレイク）】
  - Apache Iceberg形式で保存（変更履歴・タイムトラベルに対応）
  - 日付とユーザーIDでパーティション分割（検索を高速化）
  - 暗号化して保存（SSE-S3）
      │
      ▼
【Athena（分析エンジン）】
  - SQL でクエリ可能
  - 「直近N件取得」はアプリからもリアルタイムで呼び出し
```

### 5-1. オフライン時の動き

スマートフォンがオフライン（電波なし）のときでも入力できます。

```
オフライン中 → 入力データをスマートフォン内（IndexedDB）に一時保存
      ↓
オンライン復帰 → 自動的に溜まったデータを順番に送信
```

## 6. データのチェック（バリデーション）

サーバー側（Lambda）で以下のチェックを行い、不正なデータは保存されません。

| チェック項目 | ルール | エラー時 |
|------------|--------|---------|
| スコア値（疲労感・気分・やる気） | 0〜100の整数であること | 400エラーを返す |
| メモ | 280文字以内であること | 400エラーを返す |
| フラグ値 | 0〜63の整数であること（6ビットのビットマスク） | 400エラーを返す |
| ユーザーID | JWTから自動取得・UUID形式で正規表現検証（SQLインジェクション対策） | 401エラーを返す |
| カスタム項目の値 | 型（数値・テキスト・真偽値）が定義と一致すること | 400エラーを返す |
| 認証トークン | API呼び出しごとに有効期限・署名を検証 | 401エラーを返す |

> **SQLインジェクション対策**: Athenaに渡すクエリに含まれるユーザーIDは、`^[0-9a-f-]{36}$` という正規表現で事前にフォーマットを検証します。これにより、悪意のある文字列を注入してデータを改ざんする攻撃を防いでいます。

## 7. セキュリティ要件まとめ

| 脅威 | 対策 |
|------|------|
| 盗聴（通信の傍受） | すべての通信をHTTPS（TLS 1.2以上）で暗号化 |
| なりすまし（不正ログイン） | Cognito JWT（RS256署名）でAPIアクセスを制御 |
| 他人のデータへのアクセス | ユーザーIDはJWTから取得（ユーザーが自称できない） |
| 不正な値の送信 | Pydanticによるサーバー側バリデーション |
| SQLインジェクション | ユーザーIDをUUID正規表現で検証してからクエリに使用 |
| データの改ざん（保存後） | Icebergのスナップショット機能で変更履歴を保持 |
| 保存データの漏洩 | S3 SSE-S3（保存時暗号化） |
| 不正なAPIアクセス元 | CORS設定でAmplifyドメインからのみ許可 |
| Lambdaの過剰権限 | Firehoseへの書き込みとAthena実行のみに限定 |
| シークレットの漏洩 | VAPID鍵・APIキーはGitHub Secrets / SSM Parameter Storeにのみ保存 |

## 8. 非機能要件

| 項目 | 要件・現状 |
|------|-----------|
| 可用性 | AWSマネージドサービスのSLA（99.9%以上）に準拠 |
| レスポンスタイム | 記録送信: p99 < 1秒（Lambdaウォームアップ後）、直近データ取得: 最大10秒（Athenaクエリのため） |
| スケーラビリティ | Lambda同時実行数1,000まで自動スケール（個人利用では上限到達なし） |
| データ保持期間 | 無期限 |
| バックアップ | Icebergスナップショット（自動）、S3バージョニング |
| 監視 | CloudWatch でLambdaエラーを監視 |
| ログ | Lambda実行ログをCloudWatch Logsに30日保持（個人情報の過剰出力はしない） |
| コスト | 個人利用レベルでほぼ無料枠内（月額500円以下の見込み） |

## 9. 保存されるデータの構造

1レコード = 1回の記録。以下の情報がセットで保存されます。

| フィールド | 内容 | 例 |
|-----------|------|-----|
| id | 自動採番の一意ID | `550e8400-e29b-...` |
| user_id | CognitoのユーザーID | `a1b2c3d4-...` |
| record_type | 記録の種類 | `daily` / `event` / `status` |
| fatigue_score | 疲労感スコア | `70` |
| mood_score | 気分スコア | `60` |
| motivation_score | やる気スコア | `50` |
| flags | 行動フラグ（ビットマスク） | `8`（運動のみON） |
| custom_fields | カスタム項目の値（JSON） | `[{"item_id":"headache",...}]` |
| note | メモ | `"今日は頭が痛い"` |
| recorded_at | ユーザーが指定した記録日時 | `2026-03-14T09:00:00+09:00` |
| written_at | サーバーが受信した日時（UTC） | `2026-03-14T00:00:05Z` |
| timezone | タイムゾーン | `Asia/Tokyo` |
| device_id | 端末識別子 | ブラウザのUser-Agent |
| app_version | アプリバージョン | `1.0.0` |
| dt | 日付パーティション | `2026-03-14` |

### record_type の使い分け

| 種類 | 意味 | 代表例 |
|------|------|--------|
| `daily` | 疲労感・気分・やる気をまとめた日次記録 | 体調記録フォームから送信 |
| `event` | 一瞬の出来事 | 運動した、アルコール摂取 |
| `status` | 継続中の状態 | 頭痛がある、勤務中 |

### flags ビットマスクの意味

`flags` は複数の状態を1つの数値で表します。

| ビット | 値 | 意味 |
|-------|----|------|
| bit0 | 1 | 睡眠不足 |
| bit1 | 2 | 頭痛 |
| bit2 | 4 | 腹痛 |
| bit3 | 8 | 運動 |
| bit4 | 16 | アルコール |
| bit5 | 32 | カフェイン |

例: `flags = 9`（= 8 + 1）の場合 → 「運動した、かつ睡眠不足」を意味します。

> 現在はフロントエンドから常に `flags = 0` で送信しており、クイックイベント・ステータス項目は `custom_fields` で記録しています。`flags` フィールドは将来の後方互換性のために残しています。

## 10. インフラ構成の概要（非技術者向け）

| コンポーネント | 役割 | AWSサービス |
|--------------|------|-----------|
| アプリ配信 | ビルド済みアプリをスマホに届ける | Amplify Hosting |
| 認証 | ログイン・トークン発行 | Cognito |
| APIの入口 | リクエストを受け付けてLambdaに渡す | API Gateway |
| サーバー処理 | データ検証・保存処理（サーバーレス） | Lambda |
| データ配送 | バッファして安全にS3に流す | Firehose |
| データ保管 | 全記録の永続保管 | S3 Tables（Iceberg形式） |
| データ目録 | 「どんなカラムがあるか」を管理 | Glue Data Catalog |
| データ分析 | SQLでクエリ | Athena |
| インフラ定義 | AWSリソースをコードで管理 | Terraform |
| CI/CD | コード変更の自動テスト・デプロイ | GitHub Actions |

## 11. CI/CD（自動テスト・デプロイ）の流れ

コードを変更してGitHubに反映（push）すると以下が自動で走ります。

```
コード変更 → GitHub push
      ↓
【自動テスト (CI)】
  - Lambdaのユニットテスト（バリデーションロジック等）
  - フロントエンドのビルド確認・型チェック
  ↓（テスト通過後）
【自動デプロイ (CD)】
  - Lambdaのコードをzip化 → S3にアップロード
  - Terraformでインフラ変更を適用
  - AmplifyがGitHubと連携して自動ビルド・配信
```

**`terraform apply`（インフラ変更の実行）は必ず人間が確認してから実行します。自動実行は禁止としています。**
