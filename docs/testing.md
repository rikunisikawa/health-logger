## 1. バックエンドテスト（Lambda ユニットテスト）

### このテストが必要な理由

API（サーバー側の処理）は、画面を見ただけでは動作確認できない部分です。
「正しいデータが保存できるか」「不正なアクセスを弾けるか」「入力ミスを適切にエラーにするか」
といった振る舞いを、コードレベルで自動的に検証します。

特に重要なのはセキュリティの確認です。「他のユーザーのデータを覗き見・削除できないこと」は
画面テストでは見えにくいため、バックエンドテストで専用のケースを設けています。

---

### delete_record（5件）― 記録の削除

| # | 確認内容 |
|---|---------|
| 1 | 正しい操作で記録を削除できる |
| 2 | 存在しない記録への削除リクエストを適切に処理する |
| 3 | ログインせずにアクセスすると 401 エラーを返す |
| 4 | 不正な形式の ID を指定すると 400 エラーを返す |
| 5 | 他のユーザーの記録を削除しようとしても拒否される（セキュリティ確認） |

> **テスト設計の意図**: 削除は取り消せない操作のため、権限チェックを最も厳しく確認しています。
> 他ユーザーのデータへのアクセス遮断は、アプリの信頼性に直結する最重要項目です。

---

### get_env_data_latest（7件）― 環境データの取得

| # | 確認内容 |
|---|---------|
| 1 | 気温・湿度・気圧などの環境データを正常に取得できる |
| 2 | ログインせずにアクセスすると 401 エラーを返す |
| 3〜7 | 各種入力バリデーション（不正なパラメータへの対応） |

> **テスト設計の意図**: 外部データソースからの取得処理は失敗しやすい箇所のため、
> 正常系だけでなく認証エラーや不正入力のケースも網羅しています。

---

### get_item_config（3件）― 設定項目の取得

| # | 確認内容 |
|---|---------|
| 1 | 登録済みの設定項目を正常に取得できる |
| 2 | ログインせずにアクセスすると 401 エラーを返す |
| 3 | 設定がまだ登録されていない場合は空の一覧を返す |

> **テスト設計の意図**: 「設定なし」の初期状態を正しく返せるかは、
> 新規ユーザーの初回利用体験に関わる重要な動作です。

---

### get_latest（6件）― 直近の記録の取得

| # | 確認内容 |
|---|---------|
| 1 | 直近の体調記録を正常に取得できる |
| 2 | ログインせずにアクセスすると 401 エラーを返す |
| 3〜4 | 取得件数パラメータのバリデーション（不正な値への対応） |
| 5〜6 | バックエンドでの問い合わせ処理（ポーリング）が正しく動作する |

> **テスト設計の意図**: 記録の取得は毎回の利用で必ず通る処理です。
> また、データベースへの問い合わせには時間がかかる場合があるため、
> 待機・再試行の動作も自動テストで検証しています。

---

### save_item_config（9件）― 設定項目の保存

| # | 確認内容 |
|---|---------|
| 1 | 「フォーム」モードの設定項目を保存できる |
| 2 | 「イベント」モードの設定項目を保存できる |
| 3 | 「ステータス」モードの設定項目を保存できる |
| 4 | 不正な type を指定すると 400 エラーを返す |
| 5 | 不正な mode を指定すると 400 エラーを返す |
| 6 | label を未設定で送信すると 400 エラーを返す |
| 7 | item_id を未設定で送信すると 400 エラーを返す |
| 8 | ログインせずにアクセスすると 401 エラーを返す |
| 9 | 追加の入力バリデーション確認 |

> **テスト設計の意図**: 設定保存はユーザーが自由に記録項目をカスタマイズする機能です。
> 3種類のモードすべてが正しく保存できること、および不正な入力を確実に弾くことを確認しています。

## 2. フロントエンド E2E テスト（Playwright）

### このテストが必要な理由

E2E（End-to-End）テストは、実際のブラウザを自動で操作して画面の動作を確認するテストです。
「ボタンを押したら正しい画面遷移が起きるか」「入力した内容が正しく表示されるか」といった
ユーザー視点の動作を検証します。

API 呼び出し部分はモック（ダミーの返答）に差し替えているため、
本番データを誤って書き換える心配はありません。

また、モバイル（Pixel 5 サイズ）のブラウザで動作確認しているため、
スマートフォンでの表示崩れも自動的に検出できます。

---

### 前提条件

- 実際の Cognito アカウントでログインしてセッションを確立したうえでテストを実行
- API 呼び出しはモックで差し替え（本番データへの影響なし）
- Pixel 5 サイズ（モバイル画面）のブラウザで動作確認

---

### 01-display（3件）― 画面の初期表示確認

| # | 確認内容 |
|---|---------|
| 1 | スライダー3本（疲労感・気分・やる気）が画面に表示される |
| 2 | ステータスボタン4件（頭痛・腹痛・眠い・勤務中）が表示される |
| 3 | クイックイベントボタン4件（睡眠不足・運動・アルコール・カフェイン）が表示される |

> **テスト設計の意図**: アプリの基本コンポーネントが欠けていると何も記録できません。
> 画面を開いた直後に必要な要素が揃っているかをまず確認します。

---

### 02-status（3件）― ステータス ON/OFF の動作確認

| # | 確認内容 |
|---|---------|
| 4 | ステータスボタンをクリックすると ON になる（黄色に変わり「ON」ラベルが出る） |
| 5 | ON のボタンを再クリックすると OFF に戻る |
| 6 | ページをリロードしても ON の状態が保たれる（永続化の確認） |

> **テスト設計の意図**: ステータス（「現在頭痛がある」など）は、記録ボタンを押すまで
> 継続して ON にしておく必要があります。リロードで状態が消えると、記録漏れにつながるため、
> localStorage への永続化が正しく機能しているかを確認しています。

---

### 03-health-form（4件）― 体調記録フォームの動作確認

| # | 確認内容 |
|---|---------|
| 7 | 「記録する」ボタン押下後に「記録しました」の通知（トースト）が表示される |
| 8 | メモ欄に文字を入力すると文字数カウントが更新される（例: 0/280 → 5/280） |
| 9 | 過去の日時を選択すると「過去日時で記録中」のバッジが表示される |
| 10 | 「現在時刻に戻す」ボタンを押すとバッジが消える |

> **テスト設計の意図**: 記録の送信・フィードバック表示・過去日時入力は
> 日常的に使う中核機能です。「ボタンを押したが本当に記録できたのか分からない」
> という不安をなくすため、成功時の通知が確実に表示されることを確認しています。

---

### 04-toast（1件）― トースト通知の表示位置確認

| # | 確認内容 |
|---|---------|
| 11 | ページをスクロールした後に記録しても、「記録しました」の通知は画面の最上部に固定表示される |

> **テスト設計の意図**: スクロール後にトーストが画面外に消えてしまうと、
> ユーザーが記録成功に気づけません。通知が常にビューポートの上部に固定されることを確認します。

---

### 05-item-config（3件）― 設定画面の動作確認

| # | 確認内容 |
|---|---------|
| 12 | 設定ボタン（⚙️）をタップすると設定画面が開く |
| 13 | 新規追加フォームを開くと「追加」ボタンが表示される（「更新」ボタンは出ない） |
| 14 | 既存項目の編集では「更新」ボタンが表示され、記録モードに「ステータス」が含まれる |

> **テスト設計の意図**: 追加と編集では表示されるボタンが異なります。
> 誤ったボタンが表示されると「追加のつもりが上書きになる」などの操作ミスを招くため、
> 状況に応じた正しいボタンが出ているかを確認しています。

## テストの実行方法

### バックエンドテスト（Lambda）

```bash
# リポジトリのルートで実行
pytest lambda/ -v

# Lambda 関数ごとに個別実行したい場合
pytest lambda/create_record/ -v
pytest lambda/get_latest/ -v
pytest lambda/delete_record/ -v
pytest lambda/save_item_config/ -v
pytest lambda/get_item_config/ -v
pytest lambda/get_env_data_latest/ -v
```

### フロントエンド E2E テスト（Playwright）

```bash
# frontend ディレクトリで実行
cd frontend
npx playwright test

# テスト結果をブラウザで確認したい場合
npx playwright show-report
```

> **注意**: E2E テストの実行には、`.env.test` に Cognito の認証情報（テスト用アカウント）を
> 設定する必要があります。シークレット値はリポジトリには含まれていません。
> 設定方法は開発担当者に確認してください。

## CI（自動テスト）との関係

GitHub にコードをプッシュすると、上記のテストが自動で実行されます。
テストが1件でも失敗すると、本番環境へのデプロイが止まる仕組みになっています。

```
コードを GitHub にプッシュ
  ↓
バックエンドテスト（30件）を自動実行
フロントエンドビルドを自動確認
  ↓
全件通過 → 本番デプロイへ進む
失敗あり → デプロイを停止し、開発者に通知
```

これにより、バグが本番環境に入り込むリスクを自動的に抑制しています。
