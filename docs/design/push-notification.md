# プッシュ通知機能設計書

**目的**: 毎日決まった時間にプッシュ通知でリマインドし、ユーザーがタップするとアプリが開いて即座に記録できる環境を提供する。

**制約**: 既存の HealthForm コンポーネントを大きく変更しない、通知設定画面は1ページ以内。

---

## 1. システムフロー

```
ユーザー設定で時刻指定（例：朝8時）
         ↓
Service Worker で定期的に確認（深夜1回 / 日）
         ↓
指定時刻を過ぎていて、その日まだ記録がなければ通知送信
         ↓
ユーザーが通知をタップ
         ↓
アプリが前景に来る＆ HealthForm セクションにスクロール＆フォーカス
         ↓
即座にスライダーを操作して「記録する」をクリック
```

### 実装候補
- **Notification API**: ネイティブ通知（PWA/モバイルで表示）
- **Service Worker**: バックグラウンド通知チェック
- **IndexedDB**: 日次記録キャッシュ（その日記録済みかチェック用）

## 2. 画面設計

### 2.1 通知設定画面（Settings Page）

**レイアウト**: モーダルまたは独立ページ（HealthForm と兼用しない）
**配置**: メインアプリの右上 or ドロワーメニューに「⚙️ 設定」ボタンを追加

#### セクション A: 通知の有効化
```
┌─────────────────────────────────────────┐
│ 📢 プッシュ通知                          │
├─────────────────────────────────────────┤
│ ☑ 通知を有効にする                      │
│                                         │
│ ℹ️  有効にするとブラウザから許可を      │
│    求められます。許可すると毎日、      │
│    設定した時刻に体調記録のリマインド  │
│    が届きます。                         │
└─────────────────────────────────────────┘
```

**要素**:
- **トグルスイッチ**: 「通知を有効にする」（チェックボックスまたは toggle button）
- **説明文**: ブラウザ許可・プライバシーについて簡潔に記載
- **初期状態**: OFF（ユーザーが明示的に有効化）

---

#### セクション B: 通知時刻設定

```
┌─────────────────────────────────────────┐
│ 🕐 リマインド時刻                        │
├─────────────────────────────────────────┤
│ 時刻: │ 08 │ : │ 30 │                   │
│                                         │
│ ℹ️  この時刻に毎日通知が届きます。     │
│    アプリが開いていなくてもOKです。    │
└─────────────────────────────────────────┘
```

**要素**:
- **時刻入力**: 時・分のスピナーまたは `<input type="time">`
  - デフォルト: `08:30` （朝8時半）
  - 範囲: `00:00` ～ `23:59`
- **説明文**: 毎日の同じ時刻に送信されることを明記
- **プレビュー**: 「次の通知予定時刻: 明日 2026-03-22 08:30」（計算式で動的表示）

---

#### セクション C: 通知テスト＆管理

```
┌─────────────────────────────────────────┐
│ 🧪 通知テスト                            │
├─────────────────────────────────────────┤
│ ┌─────────────────────────────────────┐ │
│ │ テスト通知を送信する                 │ │
│ └─────────────────────────────────────┘ │
│ ℹ️  今すぐテスト通知を受け取ります。   │
│                                         │
│ ┌─────────────────────────────────────┐ │
│ │ 登録を解除する                       │ │
│ └─────────────────────────────────────┘ │
│ ⚠️  通知の購読を完全に解除します。     │
│    後から再度有効にできます。           │
└─────────────────────────────────────────┘
```

**要素**:
- **「テスト通知を送信する」ボタン**: すぐに通知を1件送信（ユーザーが正しく設定できたか確認用）
  - クリック後、1秒以内に通知が表示されることを期待
- **「登録を解除する」ボタン**: 通知購読を削除（赤 / danger スタイル）
  - 削除後、「通知を有効にする」トグルも OFF 状態に戻る

## 3. ユーザーフロー（通知をタップ後）

```
[ユーザーが通知をタップ]
        ↓
Service Worker が `notificationclick` イベントを受信
        ↓
focus() で既に開いているアプリをフォーカス、または新規タブで開く
        ↓
アプリが起動（App.tsx のルーティング）
        ↓
HealthForm コンポーネント存在する画面にナビゲート
        ↓
JavaScript で：
  1. HealthForm セクションまでスムーズスクロール
  2. 最初のスライダー（疲労感）にフォーカス
  3. 最初のスライダー値をハイライト（視覚的フィードバック）
        ↓
ユーザーがスライダーを操作 → 「記録する」ボタンをクリック
        ↓
記録完了 → トーストで「記録しました！」と表示
        ↓
（オプション）1.5秒後に自動で閉じる（PWA standalone モード）
```

## 4. 通知タップ時の実装ガイドライン

### 4.1 Service Worker のイベントハンドラー

```javascript
// 疑似コード：SW でしやるべき処理
self.addEventListener('notificationclick', (event) => {
  event.notification.close()

  // 1. アプリがすでに開いていればフォーカス
  event.waitUntil(
    clients.matchAll({ type: 'window', includeUncontrolled: true })
      .then((clientList) => {
        const appClient = clientList.find(c => c.url.includes('/'))
        if (appClient) {
          appClient.focus()
          // ここで postMessage を使って HealthForm へスクロール指示
          appClient.postMessage({
            type: 'NOTIFICATION_CLICKED',
            action: 'scrollToHealthForm'
          })
        } else {
          // アプリが開いていなければ新規タブで開く
          return clients.openWindow('/')
        }
      })
  )
})
```

### 4.2 フロントエンド（App.tsx）での処理

- `navigator.serviceWorker.controller` から `message` イベントをリッスン
- `NOTIFICATION_CLICKED` を受信したら、`useRef` で HealthForm DOM にアクセス
- `scrollIntoView({ behavior: 'smooth' })` でスクロール
- 最初のスライダー input にフォーカス（`element.focus()`）

## 7. 技術上の注意点

### 7.1 既存 HealthForm への影響

- ✅ スライダー・ボタン・フォーム要素は変更しない
- ✅ 通知設定は**独立した UI** として実装（settings modal または page）
- ✅ Service Worker の postMessage で通信（コンポーネント間の結合度を低く保つ）

### 7.2 ブラウザ互換性

| ブラウザ | Notification API | Service Worker | 備考 |
|---------|------------------|-----------------|------|
| Chrome  | ✅ | ✅ | フル対応 |
| Firefox | ✅ | ✅ | フル対応 |
| Safari  | ✅ | ⚠️ | iOS は限定的 |
| Edge    | ✅ | ✅ | フル対応 |

- **iOS Safari**: Service Worker 通知スケジュール非対応 → Web Push API のバックアップフローを検討
- **HTTPS 必須**: Notification API 使用には HTTPS が必須（Amplify Hosting で自動対応）

### 7.3 プライバシー・セキュリティ

- 通知設定は **localStorage** または **IndexedDB** に保存（サーバー送信不要）
- ユーザーの時刻選択は暗号化不要（local data）
- VAPID 秘密鍵が必要な場合は、AWS Secrets Manager または GitHub Secrets で管理

## 10. 後続・拡張案（Future Scope）

### 10.1 ユーザーが記録した場合の通知抑制
- その日すでに記録があれば、通知を送らない（IndexedDB チェック）

### 10.2 複数時刻設定
- 「朝 08:30」「夜 22:00」のように複数設定可能

### 10.3 通知頻度のカスタマイズ
- 「毎日」「平日のみ」「週末のみ」など

### 10.4 Web Push API との連携（サーバー側）
- Service Worker での定期送信から、バックエンド Lambda での VAPID Payload 送信に移行可能
