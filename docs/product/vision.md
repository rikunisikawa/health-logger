# Health Logger — プロダクトビジョン・戦略ドキュメント

> **作成日**: 2026-04-11  
> **最終更新**: 2026-04-11  
> **このドキュメントの目的**: 中長期の方向性を定め、実装判断の拠り所とする。

---

## 1. ミッション

> **「自分の体を、データで理解する」**
>
> 主観的な体調記録を継続的に蓄積し、客観的なデータと組み合わせることで、
> 体調の傾向・原因・改善策を自分自身で発見できる環境をつくる。

---

## 2. 現在地（2026年4月時点）

### 何ができるか

- 疲労感・気分・やる気を0〜100のスライダーで1日複数回記録
- 頭痛・腹痛・眠気などのステータスをワンタップで記録
- 運動・飲酒・カフェイン摂取などのクイックイベントを記録
- カスタム項目で自分だけの記録軸を追加
- PWA としてスマートフォンのホーム画面から即起動（オフライン記録も可）
- 蓄積データを Athena でクエリ可能（SQL ベース）

### 技術スタック

```
React+TS (Amplify) → API Gateway → Lambda (Python 3.13)
                                        ↓
                                   Firehose
                                        ↓
                              S3 Tables (Iceberg) ← Glue
                                        ↓
                                     Athena
```

### 現状の課題・限界

| 課題 | 内容 |
|------|------|
| データの「見せ方」がない | 蓄積はできるが、ダッシュボードが貧弱。傾向が見えない |
| 記録が「入力だけ」 | フィードバックがない。記録した意味を実感しにくい |
| 外部データとの接続がない | 睡眠・気圧・気温など、体調に影響するデータが孤立 |
| 分析は SQL 手動 | 非技術者には使いにくい。自動的な洞察がない |
| 個人利用のみ | マルチユーザーを想定しない設計が随所に残っている |

## 3. ビジョン（3年後の姿）

**記録するだけで、体調の「なぜ」がわかるアプリ。**

```
今日の体調が悪いのは、昨日の睡眠が浅かったから？
それとも今週ずっと雨が続いているから？
先週の運動不足？カフェインの飲みすぎ？

→ AI が過去データと外部データを突き合わせて、
  「おそらく低気圧の影響です。あなたは気圧が下がると
   疲労感スコアが平均15ポイント上昇する傾向があります」
  と教えてくれる。
```

### 3つの成長軸

| 軸 | 内容 | 現在地 |
|----|------|--------|
| **① データ蓄積の深化** | 記録できる項目を増やし、入力の継続性を高める | 基礎実装済み |
| **② 分析・可視化** | 傾向グラフ・週次レポート・相関分析をアプリ内で提供 | ほぼ未着手 |
| **③ AI アドバイス + 外部連携** | ウェアラブル・天気・医療データを統合してパーソナル AI を実現 | 未着手 |

## 4. ターゲットユーザー仮説

現在は個人利用（1名）。将来の拡張を考えるうえでの仮説ペルソナ。

### Persona A: 「体調管理を習慣化したい会社員」

- **年齢・属性**: 30代、デスクワーク中心、在宅勤務あり
- **課題**: 慢性疲労・頭痛が続くが原因不明。医者に行くほどでもない
- **目的**: 体調のパターンを把握して、仕事の配分や生活習慣を改善したい
- **行動特性**: スマートフォン中心。毎日の習慣として使えるシンプルさを求める
- **技術リテラシー**: 高くない（SQL は使えない）

### Persona B: 「健康データオタク」

- **年齢・属性**: 30〜40代、エンジニアや研究者
- **課題**: Fitbit・Apple Watch のデータは持っているが主観と紐づける手段がない
- **目的**: 主観スコアと客観データ（HRV・睡眠スコア・歩数）の相関を見たい
- **行動特性**: API・CSV エクスポート・グラフ可視化に慣れている
- **技術リテラシー**: 高い

### Persona C: 「自律神経・持病を抱える人」

- **年齢・属性**: 幅広い年代。頭痛持ち・気圧敏感・疲れやすい体質
- **課題**: 「いつ・どんな条件で」体調が崩れるかわからない。予防したい
- **目的**: トリガーを特定して、悪化する前に対処したい
- **行動特性**: 毎日記録する動機がある。AI の洞察への期待が高い

> **現時点の判断**: 特定ペルソナに絞らず、まず自分（Persona B 寄り）で使い続けてデータを貯める。
> ユーザー拡張は分析・アドバイス機能が充実してから検討する。

## 5. ロードマップ

### Phase 1: 分析・可視化の充実（〜2026年末）

**テーマ**: 記録したデータを「見て理解できる」ようにする

| 機能 | 概要 | 優先度 |
|------|------|--------|
| 週次サマリーダッシュボード | 疲労感・気分・やる気の7日推移グラフ | 高 |
| 記録の閲覧・検索 | 過去ログを日付・スコアで検索 | 高 |
| 傾向グラフ（月次・週次） | スコアの推移を折れ線グラフで可視化 | 高 |
| 相関分析（シンプル版） | 「運動した日は翌日のやる気が高い」等の傾向を表示 | 中 |
| 週次レポート通知 | 毎週月曜に先週の体調サマリーをプッシュ通知 | 中 |
| データエクスポート | CSV / JSON でダウンロード | 低 |

### Phase 2: AI アドバイス機能（2027年前半）

**テーマ**: データから「行動につながる洞察」を提供する

| 機能 | 概要 | 前提条件 |
|------|------|---------|
| 体調予測 | 過去パターンから翌日・翌週の体調スコアを予測 | 3ヶ月以上のデータ蓄積 |
| 異常検知アラート | 平均から大きく外れた体調を検出して通知 | 傾向グラフの実装後 |
| 洞察レポート | 「あなたの疲労感は〇〇と相関が高い」を週次でレポート | AI API 連携 |
| 改善アドバイス | 体調が悪い日のパターンから生活改善のヒントを提案 | 洞察レポート後 |

### Phase 3: 外部データ統合（2027年後半〜）

**テーマ**: 主観データと客観データを繋いで「体の全体像」を把握する

| 統合データ | 連携方法 | 提供価値 |
|-----------|---------|---------|
| 気象データ（気圧・気温・湿度） | 気象庁API / OpenWeatherMap | 気圧変化と体調の相関を自動検出 |
| Fitbit / Apple Watch | OAuth 認証 + Health API | 睡眠スコア・心拍数・歩数と体調の相関 |
| Google Fit | OAuth + Fitness API | Android ユーザー向けウェアラブル連携 |
| カレンダー（仕事量） | Google Calendar API | 業務負荷と体調疲労の相関 |
| 医療記録（将来） | PHR 標準 / HL7 FHIR | 服薬・通院記録との突き合わせ |

## 6. KPI（成功指標）

### 個人利用フェーズ（現在〜Phase 1）

| KPI | 目標 | 測定方法 |
|-----|------|---------|

## 7. 技術的方向性

### データ基盤の強化

現在の Athena クエリベースから、より高速でアプリ内表示に適したアーキテクチャへ。

```
現在:
  Lambda → Athena (3〜8秒) → フロントエンド

目標:
  Lambda → Athena → DynamoDB キャッシュ (< 100ms) → フロントエンド
           ↑ バッチ集計（日次 / 週次）を EventBridge でスケジュール実行
```

### AI 基盤

- **Phase 2**: Amazon Bedrock（Claude API）または OpenAI API を Lambda から呼び出し
- プロンプトは体調コンテキスト（直近30日のスコア推移）+ ユーザーの記録パターンを含める
- プライバシー: 外部 AI API に渡すデータは匿名化（user_id を除く）

### マルチユーザー化

現在の設計は user_id で完全分離されているため、インフラ変更なしに対応可能。
Cognito のユーザープール拡張 + Amplify の認証フロー調整のみで複数ユーザー対応できる。

## 8. やらないこと（Non-goals）

| やらないこと | 理由 |
|-------------|------|
| 医療診断・治療提案 | 医療機器承認が必要。法的リスクが高い |
| SNS / ソーシャル機能 | 体調データのシェアはプライバシーリスクが高い |
| リアルタイム同期（複数端末間） | Firehose の設計上、60秒バッファが入る。現状許容範囲 |
| Windows / macOS ネイティブアプリ | PWA で十分。コスト対効果が低い |
| 完全な無料プラン（他ユーザー向け） | AWS コストが発生する。最初は招待制で検討 |

## 9. 意思決定の原則

1. **データ継続性を最優先**: 壊れない・消えないことが第一。新機能より安定性
2. **入力コストを最小化**: 記録が面倒になると使わなくなる。1タップで記録できる設計を守る
3. **プライバシー by Design**: 健康データは要配慮個人情報。外部連携は明示的な同意を前提に
4. **シンプルな技術スタック**: フルサーバーレスを維持。管理コストを最小化
5. **自動化で持続可能に**: 開発・運用を Claude Code + AI で効率化し、個人で維持できる範囲を守る
