# コスト管理

> AWS サーバーレス構成の課金体系と、コストを抑えるための設計・運用指針。

---

## 1. 現在のコスト構造（概算）

個人利用（1ユーザー、1日数回の記録）を前提とした試算。

### 月間コスト概算

| サービス | 課金単位 | 想定使用量/月 | 概算コスト/月 |
|---------|---------|------------|-------------|
| **Lambda** | リクエスト数 + 実行時間 | ~300回 | ほぼ無料（無料枠内）|
| **API Gateway** | リクエスト数 | ~300回 | $0.01 以下 |
| **Kinesis Firehose** | データ転送量 | ~1MB | $0.01 以下 |
| **S3 Tables (Iceberg)** | ストレージ + API 呼び出し | ~10MB | $0.01 以下 |
| **Athena** | スキャンデータ量 | ~50MB/月 | $0.01 以下 |
| **Cognito** | アクティブユーザー数 | 1ユーザー | 無料（50,000ユーザーまで） |
| **Amplify Hosting** | ビルド時間 + 配信量 | 数回のデプロイ | $0.01 以下 |
| **CloudWatch Logs** | ログ保存量 | ~10MB | ほぼ無料 |
| **合計** | | | **~$1 以下/月** |

> ⚠️ **注意**: 上記は個人利用の概算。ユーザー数・利用頻度が増えると比例して増加する。

## 2. 各サービスの課金体系と注意点

### Lambda

```
無料枠（毎月リセット）:
  - リクエスト数: 100万回/月 まで無料
  - 実行時間: 40万 GB-秒/月 まで無料

超過後:
  - リクエスト: $0.20 / 100万リクエスト
  - 実行時間: $0.00001667 / GB-秒

現状の設定（メモリ 128MB、タイムアウト 30秒）:
  → 1日10回記録でも年間3,650回 → 無料枠の 0.37% しか使わない
```

### Athena

```
課金体系: スキャンしたデータ量 $5.00 / TB

コスト最適化の設計（実装済み）:
  ✅ Iceberg パーティション（dt カラムで日付パーティション）
     → WHERE dt >= '2026-01-01' で不要なデータを読まない
  ✅ 必要なカラムのみ SELECT（SELECT * を避ける）
  ✅ user_id による絞り込み

⚠️ コスト増加リスク:
  - SELECT * のような全カラム取得
  - WHERE 条件なしのクエリ（テーブル全体をスキャン）
  - ダッシュボードで大量のデータを一度に取得
```

### Firehose

```
課金体系: 受信データ量 $0.029 / GB

現状:
  1レコード ≈ 500バイト
  1日10件記録 → 5KB/日 → 0.15MB/月
  → 月額 $0.000004 程度（実質タダ）

バッファ設定（現在）:
  - サイズ: 5MB 以上でフラッシュ
  - 時間: 300秒（5分）でフラッシュ
  → 個人利用では時間ベースでフラッシュされる
```

### S3

```
課金体系:
  - ストレージ: $0.025 / GB / 月
  - API 呼び出し: $0.0004 / 1,000 PUT

Athena 結果ファイルに注意:
  - Athena はクエリ結果を S3 に保存する
  - 削除しないと蓄積する（athena-results/ フォルダ）
  - 推奨: S3 ライフサイクルポリシーで 30日後に自動削除
```

## 3. コスト監視の設定（推奨）

### AWS Cost Explorer の設定

```
AWS マネジメントコンソール → Billing → Cost Explorer

推奨設定:
  1. 月次コストのサービス別内訳を確認
  2. 予算アラートの設定（例: $10/月を超えたらメール通知）
```

### 予算アラートの設定方法

```
AWS マネジメントコンソール → Billing → Budgets → 予算を作成
  - 予算タイプ: コスト予算
  - 予算額: $10/月（任意）
  - アラート: 80% 達成時にメール通知
```

## 4. コスト最適化の実装済み対策

| 対策 | 実装状況 | 効果 |
|-----|---------|------|
| Lambda メモリを最小設定（128MB） | ✅ | 実行コスト削減 |
| Athena に WHERE 条件を必ず付ける | ✅ | スキャン量削減 |
| Iceberg パーティション（日付別） | ✅ | スキャン量削減 |
| SELECT * を使わない | ✅ | スキャン量削減 |
| Firehose でバッチ処理 | ✅ | API 呼び出し削減 |
| Amplify CDN キャッシュ | ✅ | データ転送削減 |

## 5. コスト増加のリスクシナリオ

### シナリオ1: Athena の暴走クエリ

```
リスク: WHERE 条件なしで大量データをスキャンするクエリが実行された場合
影響: $5.00 / TB のスキャン料金が一度に発生

対策（現状）:
  - get_latest Lambda でクエリに LIMIT 100 を設定
  - user_id と recorded_at の範囲を WHERE 条件に含める

追加対策候補:
  - Athena のクエリスキャン量上限の設定
  - DynamoDB キャッシュで Athena クエリ回数を削減
```

### シナリオ2: Lambda の無限ループ

```
リスク: バグにより Lambda が異常に多く呼び出された場合
影響: 月100万回の無料枠を超えると課金発生

対策:
  - Lambda に同時実行数の上限を設定（現状未設定）
  - CloudWatch Alarm でリクエスト数の異常を検知
```

### シナリオ3: S3 ストレージの肥大化

```
リスク: Athena クエリ結果が削除されずに蓄積
影響: ストレージコストが緩やかに増加

対策候補:
  - S3 ライフサイクルポリシー（30日で自動削除）を設定
```

## 6. スケール時のコスト試算

ユーザー数が増えた場合の月額概算:

| ユーザー数 | Lambda/月 | Athena/月 | S3/月 | 合計/月 |
|----------|----------|----------|------|--------|
| 1人（現在） | $0 | $0.01 | $0.01 | **~$1** |
| 10人 | $0 | $0.10 | $0.05 | **~$1** |
| 100人 | $0 | $1.00 | $0.50 | **~$3** |
| 1,000人 | $0.50 | $10.00 | $5.00 | **~$20** |
| 10,000人 | $5.00 | $100.00 | $50.00 | **~$200** |

> Lambda の無料枠（100万リクエスト/月）は全ユーザー共通のため、
> 1,000ユーザー規模までは Lambda コストはほぼ無料。
